# 회원 도메인 — 이벤트스토밍 산출물

> 진행 중인 문서. 이벤트스토밍 → 애그리거트 설계를 단계별로 기록한다.

---

## 1단계: 이벤트스토밍 산출물

### 1-1. 이벤트 → 커맨드 → 애그리거트 매핑표

| Actor | Command | Aggregate | Event |
|-------|---------|-----------|-------|
| 사용자 | 소셜 로그인으로 가입하기 | Member | 회원이 가입됨 |
| 회원 | 정보 수정하기 | Member | 회원 정보가 수정됨 |
| 관리자 | 회원 정지하기 | Member | 회원이 정지됨 |
| 회원 | 탈퇴하기 | Member | 회원이 탈퇴됨 |
| 관리자 | 회원 탈퇴 처리하기 | Member | 회원이 탈퇴됨 |

### 1-2. 상태 전이

```
가입 → ACTIVE
ACTIVE → SUSPENDED (관리자가 정지)
ACTIVE → DEACTIVATED (본인이 탈퇴)
SUSPENDED → DEACTIVATED (관리자가 탈퇴 처리)
```

- DEACTIVATED는 최종 상태. 다른 상태로 전이 불가.
- 탈퇴 = 상태 변경 (실제 삭제 아님)

### 1-3. 규칙 분류

#### 0번 질문으로 걸러낸 것

| 규칙 | 구분 | 처리 |
|------|------|------|
| OAuth2 토큰 교환/검증 | 기술/보안 | 어댑터 계층에서 처리 |

#### 도메인 규칙 분류

| 규칙 | 분류 | 비고 |
|------|------|------|
| 이메일 중복 가입 불가 | 불변식 | 구현은 서비스에 배치 (저장소 조회 필요) |
| 이메일 형식 검증 | 값 검증 | Email VO 생성자에서 처리 |
| 닉네임 중복 불가 | 불변식 | 구현은 서비스에 배치 (저장소 조회 필요) |
| 닉네임 길이/특수문자 제한 | 값 검증 | Nickname VO 생성자에서 처리 |
| 가입 시 상태는 ACTIVE | 불변식 | 생성 불변식 |
| ACTIVE에서만 정지 가능 | 불변식 | 전이 불변식. 상태 전이의 출발 조건 |
| DEACTIVATED에서 다른 상태로 전이 불가 | 불변식 | 전이 불변식. 최종 상태 |
| ACTIVE → DEACTIVATED는 본인만 | 정책 | 전이 경로를 갈라놓는 정책. 본인 확인은 도메인 식별자 비교(requesterId == memberId)로 Member가 검증 |
| SUSPENDED → DEACTIVATED는 관리자 경로만 | 정책 | 전이 경로를 갈라놓는 정책. 관리자 권한 확인은 서비스 계층, Member는 상태 전이 규칙만 담당 |
| ACTIVE에서만 정보 수정 가능 | 정책 | 위반해도 데이터는 정상, 운영 규칙 |
| SUSPENDED 회원은 활동 불가 | 정책 | 도메인에서 상태 판별 제공, 구현은 어댑터 |

#### 규칙 분류 과정에서 도출된 설계 원칙

- **불변식 vs 정책 구분법**: 불변식은 "전이가 존재할 수 있는가"(절대 안 됨), 정책은 "이 전이를 지금 이 조건에서 해도 되는가"(조건부로 됨)
- **정책이 전이 경로를 갈라놓을 수 있다**: "본인만 탈퇴", "관리자 경로로만 탈퇴 처리"는 의미 분류상 정책이지만, 전이 경로 자체를 나눈다. 정책이라고 해서 메서드를 하나로 합쳐야 하는 것은 아니다
- **의미 분류 / 구현 배치 / 메서드 shape는 별개 결정**: 규칙의 의미 분류(정책), 실제 검사 위치(서비스 or 애그리거트), 애그리거트가 외부에 드러내는 행위 API(메서드 분리 여부)는 각각 독립적으로 판단한다
- **본인 확인 vs 관리자 확인의 비대칭**: 본인 확인은 도메인 식별자 비교(`requesterId == memberId`)이므로 Member가 검증 가능. 관리자 확인은 인증/인가 체계 의존이므로 서비스 계층에서 처리
- **전이 불변식**: 상태 전이에도 불변식이 있다. DEACTIVATED → SUSPENDED 같은 "전이 자체가 있으면 안 되는 것"
- **반례 판별법**: 모든 규칙에 반례를 만들 필요는 없다. "가입 시 상태는 ACTIVE" 같은 건 보자마자 불변식. 분류가 애매할 때 반례를 만들어 판별한다

## 2단계: 애그리거트 내부 설계 (진행 중)

### 2-1. 커맨드별 설계 워크시트

#### 커맨드: 소셜 로그인으로 가입하기

**맥락 수집 (1단계에서 가져오기)**

| 항목 | 내용 |
|------|------|
| Actor | 사용자 |
| Event | 회원이 가입됨 |
| 상태 전이 | → ACTIVE (신규 생성) |
| 관련 값 검증 | 이메일 형식 검증 |
| 관련 불변식 | 이메일 중복 가입 불가, 가입 시 상태는 ACTIVE |
| 관련 정책 | - |
| 후속 반응 | - |

**설계 도출**

| 도출 항목 | 내용 |
|-----------|------|
| 대상 식별자 | - (신규 생성) |
| 도메인 입력 | email, name (소셜 서비스에서 제공) |
| 상태 후보 | email (Email VO), nickname (Nickname VO) |
| 호출 컨텍스트 | - |
| 행위 | 회원 생성 |

**설계 결정:**
- 소셜 이름(name)은 닉네임 초기값으로만 사용. 도메인 입력이지만 Member 속성이 아님 (행위 실행 시 일시적으로 사용)
- provider, socialId는 저장하지 않음. 이메일로 회원 식별/중복 체크. 회원은 지원 도메인이므로 단순하게 가져감
- email → Email VO (형식 검증), nickname → Nickname VO (길이/특수문자 검증)

#### 커맨드: 정보 수정하기

**맥락 수집 (1단계에서 가져오기)**

| 항목 | 내용 |
|------|------|
| Actor | 회원 (본인) |
| Event | 회원 정보가 수정됨 |
| 상태 전이 | 없음 (ACTIVE 유지) |
| 관련 값 검증 | 닉네임 길이/특수문자 제한 |
| 관련 불변식 | 닉네임 중복 불가 |
| 관련 정책 | ACTIVE에서만 정보 수정 가능 |
| 후속 반응 | - |

**설계 도출**

| 도출 항목 | 내용 |
|-----------|------|
| 대상 식별자 | memberId (서비스에서 Member 조회에 사용) |
| 도메인 입력 | nickname |
| 상태 후보 | nickname (기존 속성 업데이트) |
| 호출 컨텍스트 | 본인 여부 (인증 계층에서 처리) |
| 행위 | 닉네임 변경 |

**설계 결정:**
- 수정 가능한 속성은 nickname만. email은 소셜 계정에 종속이라 고정
- 닉네임 변경 시 Nickname VO를 새로 생성하므로 값 검증이 자동 적용
- memberId는 서비스에서 Member를 찾는 데 소비. 애그리거트 메서드 파라미터에는 들어가지 않음

#### 커맨드: 회원 정지하기

**맥락 수집 (1단계에서 가져오기)**

| 항목 | 내용 |
|------|------|
| Actor | 관리자 |
| Event | 회원이 정지됨 |
| 상태 전이 | ACTIVE → SUSPENDED |
| 관련 값 검증 | - |
| 관련 불변식 | DEACTIVATED에서 다른 상태로 전이 불가 (소거법으로 ACTIVE에서만 정지 가능) |
| 관련 정책 | - |
| 후속 반응 | - |

**설계 도출**

| 도출 항목 | 내용 |
|-----------|------|
| 대상 식별자 | memberId (서비스에서 Member 조회에 사용) |
| 도메인 입력 | - (상태 전이만) |
| 상태 후보 | - (기존 status 변경) |
| 호출 컨텍스트 | 관리자 여부 (인증 계층에서 처리) |
| 행위 | 회원 정지 (상태 전이) |

**설계 결정:**
- Member.suspend()에 파라미터 없음. 자기 상태를 ACTIVE → SUSPENDED로 변경
- 정지 사유는 현재 저장하지 않음 (지원 도메인 단순화). 필요시 도메인 입력으로 추가

#### 커맨드: 탈퇴하기

**맥락 수집 (1단계에서 가져오기)**

| 항목 | 내용 |
|------|------|
| Actor | 회원 (본인) |
| Event | 회원이 탈퇴됨 |
| 상태 전이 | ACTIVE → DEACTIVATED |
| 관련 값 검증 | - |
| 관련 불변식 | DEACTIVATED에서 다른 상태로 전이 불가 |
| 관련 정책 | ACTIVE → DEACTIVATED는 본인만 (전이 경로를 갈라놓는 정책) |
| 후속 반응 | - |

**설계 도출**

| 도출 항목 | 내용 |
|-----------|------|
| 대상 식별자 | memberId (서비스에서 Member 조회에 사용) |
| 도메인 입력 | requesterId (본인 확인용, 도메인 식별자 비교) |
| 상태 후보 | - (기존 status 변경) |
| 호출 컨텍스트 | - |
| 행위 | 본인 탈퇴 — withdraw(requesterId) |

**설계 결정:**
- 본인 확인은 `requesterId == memberId` 비교로, Member가 직접 검증. 도메인 식별자 비교이므로 인증 체계 의존 아님
- requesterId는 서비스가 인증 계층에서 꺼내서 전달

#### 커맨드: 회원 탈퇴 처리하기

**맥락 수집 (1단계에서 가져오기)**

| 항목 | 내용 |
|------|------|
| Actor | 관리자 |
| Event | 회원이 탈퇴됨 |
| 상태 전이 | SUSPENDED → DEACTIVATED |
| 관련 값 검증 | - |
| 관련 불변식 | DEACTIVATED에서 다른 상태로 전이 불가 |
| 관련 정책 | SUSPENDED → DEACTIVATED는 관리자 경로만 (전이 경로를 갈라놓는 정책) |
| 후속 반응 | - |

**설계 도출**

| 도출 항목 | 내용 |
|-----------|------|
| 대상 식별자 | memberId (서비스에서 Member 조회에 사용) |
| 도메인 입력 | - (상태 전이만) |
| 상태 후보 | - (기존 status 변경) |
| 호출 컨텍스트 | 관리자 여부 (서비스/인가 계층에서 확인 후 이 메서드 호출) |
| 행위 | 관리자 탈퇴 처리 — withdrawByAdmin() |

**설계 결정:**
- Member는 "SUSPENDED에서만 이 경로가 가능하다"는 상태 전이 규칙만 담당
- "호출자가 실제 관리자인가"는 서비스 계층에서 인가 확인 후 이 메서드를 호출하는 구조로 보장

### 워크시트 통합

#### 식별성 결정

- memberId (PK, 인공키) — 애그리거트의 대표 식별자
- email (고유 제약) — 중복 가입 방지용. 식별성이 아니라 고유 속성

#### 상태 후보 통합

| 출처 워크시트 | 상태 후보 |
|---------------|-----------|
| 가입하기 | email (Email VO), nickname (Nickname VO) |
| 정보 수정하기 | nickname (업데이트) |
| 정지 / 탈퇴 / 탈퇴 처리 | status 변경 |

**→ Member가 보존하는 핵심 상태: email, nickname, status**

#### 행위 통합

| 행위 | 메서드 | 파라미터 |
|------|--------|----------|
| 회원 생성 | register() | email, nickname |
| 닉네임 변경 | changeNickname() | nickname |
| 정지 | suspend() | - |
| 본인 탈퇴 | withdraw() | requesterId |
| 관리자 탈퇴 처리 | withdrawByAdmin() | - |

#### 규칙 통합

**불변식:**
- 가입 시 상태는 ACTIVE (생성 불변식)
- DEACTIVATED에서 다른 상태로 전이 불가 (전이 불변식)

**값 검증:**
- 이메일 형식 → Email VO 생성자
- 닉네임 길이/특수문자 → Nickname VO 생성자

**정책 (전이 경로 분리):**
- ACTIVE → DEACTIVATED는 본인만 → withdraw(requesterId)
- SUSPENDED → DEACTIVATED는 관리자 경로만 → withdrawByAdmin()
- ACTIVE에서만 정보 수정 가능

**외부 협력 (서비스 배치 불변식):**
- 이메일 중복 불가 → 저장소 조회 필요
- 닉네임 중복 불가 → 저장소 조회 필요

### 2-2. 내부 구성요소 분류

| 후보 | 시간이 지나도 추적? | 독립 존재? | 같은 값이면 교환? | 분류 |
|------|---------------------|------------|---------------------|------|
| Member | O | O (루트) | X | Root Entity (애그리거트 루트) |
| email | X | X | O | Value Object → Email VO (형식 검증, 독립 도메인 용어) |
| nickname | X | X | O | Value Object → Nickname VO (길이/특수문자 검증, 독립 도메인 용어) |
| status | X | X | O | MemberStatus enum (상태 전이 로직의 핵심) |

- child entity 없음. 단순 도메인이므로 root entity + VO 2개 + enum 1개로 구성

### 2-3. 도메인 모델 요약

```
[Member] — 엔티티 (애그리거트 루트)
  식별성:
    - memberId (PK, 인공키)

  핵심 상태:
    - email (Email VO, 고유 제약)
    - nickname (Nickname VO)
    - status (MemberStatus enum: ACTIVE, SUSPENDED, DEACTIVATED)

  핵심 행위:
    - register() — 회원 생성
    - changeNickname() — 닉네임 변경
    - suspend() — 회원 정지
    - withdraw(requesterId) — 본인 탈퇴
    - withdrawByAdmin() — 관리자 탈퇴 처리

  규칙:
    - 가입 시 상태는 ACTIVE (생성 불변식)
    - ACTIVE에서만 정지 가능 (전이 불변식)
    - DEACTIVATED에서 다른 상태로 전이 불가 (전이 불변식)
    - ACTIVE 회원은 본인만 탈퇴 가능 (정책 — 전이 경로 분리, requesterId로 검증)
    - SUSPENDED 회원은 관리자 경로로만 탈퇴 처리 (정책 — 전이 경로 분리, 관리자 확인은 서비스)
    - ACTIVE에서만 정보 수정 가능 (정책)

  외부 협력 지점:
    - 이메일 중복 검증 → 저장소 조회 필요
    - 닉네임 중복 검증 → 저장소 조회 필요

  후속 반응:
    - (없음)
```

## 3단계: TDD로 코드 구현

### 3-1. 테스트 할 일 목록

2-3 도메인 모델 요약의 핵심 행위, 규칙, 외부 협력 지점에서 도출.

**VO 값 검증:**
- [ ] Email: 유효한 이메일 형식으로 생성할 수 있다
- [ ] Email: 잘못된 형식이면 생성 실패한다
- [ ] Nickname: 유효한 닉네임으로 생성할 수 있다
- [ ] Nickname: 길이/특수문자 위반이면 생성 실패한다

**생성 (register):**
- [ ] register() 후 상태는 ACTIVE이다
- [ ] register() 후 email, nickname이 세팅된다

**닉네임 변경 (changeNickname):**
- [ ] ACTIVE 회원은 닉네임을 변경할 수 있다
- [ ] ACTIVE가 아닌 상태에서 닉네임 변경하면 실패한다

**정지 (suspend):**
- [ ] ACTIVE 회원을 정지하면 SUSPENDED가 된다
- [ ] ACTIVE가 아닌 상태에서 정지하면 실패한다

**본인 탈퇴 (withdraw):**
- [ ] ACTIVE 회원이 본인 ID로 탈퇴하면 DEACTIVATED가 된다
- [ ] 본인이 아닌 ID로 탈퇴하면 실패한다
- [ ] ACTIVE가 아닌 상태에서 본인 탈퇴하면 실패한다

**관리자 탈퇴 처리 (withdrawByAdmin):**
- [ ] SUSPENDED 회원을 관리자가 탈퇴 처리하면 DEACTIVATED가 된다
- [ ] SUSPENDED가 아닌 상태에서 관리자 탈퇴 처리하면 실패한다

**도메인 테스트 범위 밖 (서비스 테스트에서 다룸):**
- 이메일 중복 검증
- 닉네임 중복 검증
- 관리자 권한 확인

## 설계 결정 기록

| 결정 | 이유 |
|------|------|
| 소셜 로그인만 지원 (이메일 가입 없음) | 회원은 지원 도메인, 단순하게 가져감. 비밀번호 관련 로직 전부 불필요 |
| 가입 즉시 ACTIVE (PENDING 없음) | 승인 절차 단순화 |
| 정지 = 영구 정지 (해제 없음) | 단순화 |
| 정지된 회원의 탈퇴는 관리자만 가능 | 본인 탈퇴 시 로그인 예외 처리 필요 → 단순화 |
| 식별자는 memberId(PK), email은 고유 제약 속성 | 애그리거트 식별은 항상 인공키. email은 중복 가입 방지용. provider/socialId 저장 안 함 |
| 소셜 이름은 닉네임 초기값으로만 사용 | 가입 과정 최소화 |
| email → Email VO, nickname → Nickname VO | 형식 검증 로직 캡슐화, 타입 안전성 |
| 수정 가능한 속성은 nickname만 | email은 소셜 계정에 종속이라 고정 |
| 탈퇴 행위를 withdraw() / withdrawByAdmin()으로 분리 | 의미 분류상 "본인만/관리자만"은 정책이지만, 출발 상태·Actor·비즈니스 용어가 모두 다르므로 메서드도 분리. 정책 분류가 메서드 shape를 결정하지 않는다 |
| 본인 확인은 Member가, 관리자 확인은 서비스가 담당 | 본인 확인은 도메인 식별자 비교(requesterId == memberId), 관리자 확인은 인증/인가 체계 의존 |

### 후순위 검토

| 검토 항목 | 현재 | 전환 시 변경점 |
|-----------|------|----------------|
| 소셜 계정 연동 테이블 분리 | 이메일로 회원 식별, provider/socialId 저장 안 함 | SocialAccount 테이블 추가 (provider + providerId 유니크), Member는 이메일 유니크 유지, 여러 소셜 계정을 한 회원에 연결 가능 |
