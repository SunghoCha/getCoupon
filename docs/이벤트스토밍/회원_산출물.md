# 회원 도메인 산출물

> 설계 과정 기록은 [회원.md](회원.md) 참고.
> 이 문서는 도메인 모델과 유스케이스의 **최종 결과**를 사람이 참조하기 위한 문서다.

---

## 1. 도메인 모델 요약

```
[Member] — 엔티티 (애그리거트 루트)

  식별성:
    - memberId (PK, 인공키)

  핵심 상태:
    - email (Email VO, 고유 제약)
    - nickname (Nickname VO)
    - status (MemberStatus enum: ACTIVE, SUSPENDED, DEACTIVATED)

  핵심 행위:
    - register() — 회원 생성
    - changeNickname() — 닉네임 변경
    - suspend() — 회원 정지
    - withdraw() — 본인 탈퇴
    - withdrawByAdmin() — 관리자 탈퇴 처리

  상태 전이:
    가입 → ACTIVE
    ACTIVE → SUSPENDED (관리자가 정지)
    ACTIVE → DEACTIVATED (본인이 탈퇴)
    SUSPENDED → DEACTIVATED (관리자가 탈퇴 처리)
    DEACTIVATED → 최종 상태. 다른 상태로 전이 불가.

  규칙:
    불변식:
      - 가입 시 상태는 ACTIVE (생성 불변식)
      - DEACTIVATED에서 다른 상태로 전이 불가 (전이 불변식)

    값 검증:
      - 이메일 형식 → Email VO 생성자
      - 닉네임 길이/특수문자 → Nickname VO 생성자

    정책 (전이 경로 분리):
      - ACTIVE → DEACTIVATED는 본인만 → withdraw(requesterId)
      - SUSPENDED → DEACTIVATED는 관리자 경로만 → withdrawByAdmin()
      - ACTIVE에서만 정보 수정 가능

    외부 협력 (서비스 배치):
      - 이메일 중복 검증 → 저장소 조회 필요
      - 닉네임 중복 검증 → 저장소 조회 필요
```

## 2. 유스케이스 워크시트

### 2.1 회원 가입 (register)

```
유스케이스: 회원 가입
목표/의도: 신규 회원을 생성한다
Actor/Trigger: 비로그인 사용자 회원가입 요청

입력: MemberRegisterRequest(email, nickname)
대상 식별: 없음 (신규 생성)
사전 조건: 없음

외부 규칙 / 교차 검증:
  - 이메일 중복 불가 (memberRepository.existsByEmail)
  - 닉네임 중복 불가 (memberRepository.existsByNickname)

도메인 행위: Member.register(Email, Nickname)

트랜잭션 경계: 회원 생성 및 저장까지 같은 트랜잭션
일관성 방식: same transaction

성공 시 관측 포인트:
  - 반환값: 생성된 회원 식별자 또는 성공 응답
  - 저장 후 상태: Member 생성됨, 상태 ACTIVE
  - 외부 부수효과: 없음 (향후 가입 이벤트/outbox 가능)

유스케이스 수준 실패:
  - 교차 검증 실패: 이메일 중복 → DuplicateEmailException
  - 교차 검증 실패: 닉네임 중복 → DuplicateNicknameException
  - 확정 실패: DB 고유키 충돌 (race condition)

권한/인증 메모: 없음
멱등성 메모: 동일 요청 재시도 정책 별도 검토
no-op 정책: 없음
어댑터 매핑 메모: race condition 시 DataIntegrityViolationException → @ControllerAdvice에서 번역 필요
```

### 2.2 닉네임 변경 (changeNickname)

```
유스케이스: 닉네임 변경
목표/의도: 회원이 자신의 닉네임을 바꾼다
Actor/Trigger: 로그인한 회원의 닉네임 변경 요청

입력: MemberNicknameChangeRequest(memberId, nickname)
대상 식별: memberId → memberRepository.findById
사전 조건: 없음

외부 규칙 / 교차 검증:
  - 닉네임 중복 불가 (memberRepository.existsByNickname)

도메인 행위: member.changeNickname(Nickname)

트랜잭션 경계: 조회, 검증, 변경, 저장까지 같은 트랜잭션
일관성 방식: same transaction

성공 시 관측 포인트:
  - 저장 후 상태: 닉네임이 변경됨 (DB 재조회)

유스케이스 수준 실패:
  - 찾기 실패: 회원 없음 → MemberNotFoundException
  - 교차 검증 실패: 닉네임 중복 → DuplicateNicknameException
  - 도메인 위임 실패: 상태상 변경 불가 → MemberStatusException
  - 확정 실패: race condition (더티 체킹이라 commit 시점에 발생)

권한/인증 메모: memberId와 인증 사용자 일치 여부가 서비스 책임이면 추가
멱등성 메모: 동일 요청 반복 시 정책 확인
no-op 정책: 동일 닉네임 요청도 성공 처리 (API 계약 — 멱등성 보장)
  구현상 중복 검증은 스킵하되 도메인 호출은 수행하여 상태 검증은 항상 거친다
어댑터 매핑 메모: race condition → @ControllerAdvice에서 번역 필요
```

### 2.3 탈퇴 (withdraw)

```
유스케이스: 회원 탈퇴 (본인)
목표/의도: 회원이 자신의 계정을 탈퇴한다
Actor/Trigger: 로그인한 회원의 탈퇴 요청

입력: Long requesterId (인증 계층에서 추출, DTO 없음)
대상 식별: requesterId → memberRepository.findById
사전 조건: 없음

외부 규칙 / 교차 검증: 없음

도메인 행위: member.withdraw()

트랜잭션 경계: 조회, 상태 변경까지 같은 트랜잭션
일관성 방식: same transaction

성공 시 관측 포인트:
  - 저장 후 상태: 상태 DEACTIVATED (DB 재조회)

유스케이스 수준 실패:
  - 찾기 실패: 회원 없음 → MemberNotFoundException
  - 도메인 위임 실패: 상태상 탈퇴 불가 → MemberStatusException

권한/인증 메모: requesterId는 인증 계층에서 추출하여 전달
멱등성 메모: 없음
no-op 정책: 없음
어댑터 매핑 메모: 없음

확정 사항 (결정 기록):
  - 초기 설계(회원.md 174행)에서는 대상 식별자(memberId)와 도메인 입력(requesterId)을 구분했으나,
    서비스 구조상 findById(requesterId)로 조회하면 member.id == requesterId가 항상 참이 되어
    "본인 확인" 분기는 서비스 API를 통해 도달할 수 없음을 확인.
  - 따라서 MemberWithdrawRequest DTO를 제거하고 void withdraw(Long requesterId)만 유지.
  - 도메인 메서드도 member.withdraw() (인자 없음)로 단순화. 본인 확인 책임은 인증 계층이 전담.
```

### 2.4 회원 정지 (suspend)

```
유스케이스: 회원 정지
목표/의도: 관리자가 회원을 정지시킨다
Actor/Trigger: 관리자의 회원 정지 요청

입력: MemberSuspendRequest(memberId)
대상 식별: memberId → memberRepository.findById
사전 조건: 없음

외부 규칙 / 교차 검증: 없음

도메인 행위: member.suspend()

트랜잭션 경계: 단일 트랜잭션
일관성 방식: same transaction

성공 시 관측 포인트:
  - 저장 후 상태: 상태 SUSPENDED (DB 재조회)

유스케이스 수준 실패:
  - 찾기 실패: 회원 없음 → MemberNotFoundException
  - 도메인 위임 실패: ACTIVE가 아닌 상태에서 정지 → MemberStatusException

권한/인증 메모: 관리자 권한 확인은 인가 계층 책임 (서비스는 권한 검증하지 않음)
멱등성 메모: 없음
no-op 정책: 없음
어댑터 매핑 메모: 없음
```

### 2.5 관리자 탈퇴 처리 (withdrawByAdmin)

```
유스케이스: 관리자 탈퇴 처리
목표/의도: 관리자가 정지된 회원을 탈퇴 처리한다
Actor/Trigger: 관리자의 회원 탈퇴 처리 요청

입력: MemberWithdrawByAdminRequest(memberId)
대상 식별: memberId → memberRepository.findById
사전 조건: 없음

외부 규칙 / 교차 검증: 없음

도메인 행위: member.withdrawByAdmin()

트랜잭션 경계: 단일 트랜잭션
일관성 방식: same transaction

성공 시 관측 포인트:
  - 저장 후 상태: 상태 DEACTIVATED (DB 재조회)

유스케이스 수준 실패:
  - 찾기 실패: 회원 없음 → MemberNotFoundException
  - 도메인 위임 실패: SUSPENDED가 아닌 상태에서 탈퇴 처리 → MemberStatusException

권한/인증 메모: 관리자 권한 확인은 인가 계층 책임 (서비스는 권한 검증하지 않음)
멱등성 메모: 없음
no-op 정책: 없음
어댑터 매핑 메모: 없음
```
