# 도메인 개발 가이드

> **주의: 이 문서는 참고용이다.**
> 회원(Member) 도메인 같은 비교적 단순한 도메인을 기반으로 작성된 가이드이며,
> letterPick의 모든 도메인에 그대로 적용하는 것이 아니다.
>
> 특히 쿠폰 발급 전략처럼 복잡한 도메인에서는 이 가이드의 전제가 맞지 않을 수 있다.
> - 하나의 애그리거트 = 하나의 상태 전이 흐름이라는 전제
> - 애그리거트 경계가 비교적 명확하다는 전제
> - 도메인 테스트에 인프라 의존이 없다는 전제
>
> 기본 틀과 사고 방식은 참고하되, 각 도메인의 특성에 맞게 판단한다.

---

> 이벤트스토밍 산출물 → 엔티티 설계 → TDD 코드 구현까지의 실전 흐름.
> 개념 설명은 최소화하고, **"다음에 뭘 하지?"에 답하는 순서도**로 구성한다.

---

## 전체 흐름

```
1. 이벤트스토밍 산출물 정리
       ↓
2. 애그리거트 내부 설계 (엔티티/값객체 결정 + 도메인 모델 요약)
   → 2단계 끝에 산출물 문서 초안 생성 (도메인 모델 요약 + 유스케이스 v1)
       ↓
3. 도메인 TDD (도메인 테스트 + 도메인 로직 구현)
       ↓
4. 산출물 문서 구체화 (도메인 TDD 경험으로 유스케이스 v2 완성)
       ↓
5. 서비스 TDD (유스케이스 워크시트 기반)
       ↓
리팩토링은 각 TDD 사이클 안에서 반복
```

### 문서는 살아있는 문서다

이 흐름은 한 방향으로 끝나지 않는다. 도메인 TDD 중에 발견한 것(새 예외, 설계 불일치, 서비스 책임 규칙 등)은 산출물 문서로 돌아와서 반영한다. 서비스 TDD 중에 도메인 설계가 어색하면 도메인으로 돌아간다. 문서가 코드와 괴리되지 않도록 변경 사항이 생기면 해당 문서를 갱신한다.

- 2단계 끝에 산출물 문서 초안을 만든다 (도메인 모델 요약 + 유스케이스 v1). 이 시점에서 아는 만큼만 적는다.
- 4단계에서 도메인 TDD 경험을 바탕으로 산출물 문서를 구체화한다 (유스케이스 실패 케이스, 외부 규칙, 서비스 분기 등 추가).
- 산출물 문서는 도메인별로 관리한다 (예: `이벤트스토밍/회원_산출물.md`).

### 문서 역할 구분

- **이벤트스토밍 과정 기록** (예: `이벤트스토밍/회원.md`) — 설계 과정의 기록. "어떻게 이 결론에 도달했는가"
- **산출물 문서** (예: `이벤트스토밍/회원_산출물.md`) — 사람이 참조하는 결과물. 도메인 모델 요약 + 유스케이스 워크시트

---

## 1단계: 이벤트스토밍 산출물 정리

이벤트스토밍이 끝나면 보드 위에 이런 것들이 붙어 있다.
이걸 **아래 표 형식으로 정리**한다.

### 1-1. 이벤트 → 커맨드 → 애그리거트 매핑표

| Actor | Command | Aggregate | Event | 비고 |
|-------|---------|-----------|-------|------|
| 사용자 | 소셜 로그인으로 가입하기 | Member | 회원이 가입됨 | |
| 회원 | 정보 수정하기 | Member | 회원 정보가 수정됨 | |
| 관리자 | 회원 정지하기 | Member | 회원이 정지됨 | |
| 회원 | 탈퇴하기 | Member | 회원이 탈퇴됨 | |
| 관리자 | 회원 탈퇴 처리하기 | Member | 회원이 탈퇴됨 | |

> **핵심**: 이 표의 Aggregate 열이 곧 **도메인 클래스 후보**다.
> Command는 애그리거트가 제공해야 할 **도메인 행위의 후보**를 준다.
> 다만 커맨드의 이름/입력이 애그리거트 메서드와 1:1 대응되지는 않는다.

### 1-2. 정책/규칙 목록

이벤트스토밍에서 나온 핫스팟, 정책을 규칙으로 정리한다.

> **각 상태 전이마다 출발 상태 조건을 명시적인 규칙으로 적는다.**
> 다른 규칙에서 소거법으로 유도되더라도 생략하지 않는다.
> 생략하면 워크시트 → 도메인 모델 요약 → 테스트로 이어지는 과정에서 빠진다.

| 규칙 | 적용 시점 | 비고 |
|------|-----------|------|
| 이메일 중복 가입 불가 | 가입 시 | |
| 이메일 형식 검증 | 가입 시 | |
| 닉네임 중복 불가 | 가입/닉네임 변경 시 | |
| 닉네임 길이/특수문자 제한 | 가입/닉네임 변경 시 | |
| 가입 시 상태는 ACTIVE | 가입 시 | |
| ACTIVE에서만 정지 가능 | 정지 시 | |
| ACTIVE에서만 정보 수정 가능 | 정보 수정 시 | |
| ACTIVE 회원은 본인 요청으로만 탈퇴 가능 | 본인 탈퇴 시 | |
| SUSPENDED 회원은 관리자 처리로만 탈퇴 가능 | 관리자 탈퇴 처리 시 | |
| DEACTIVATED에서 다른 상태로 전이 불가 | 전체 | 최종 상태 |

### 1-3. 규칙 분류

1-2에서 정리한 규칙들을 분류한다.
**의미 분류**(이 규칙이 무엇인가)와 **구현 배치**(어디에 둘 것인가)는 별개의 판단이므로 나눠서 생각한다.

#### 0번 질문: 도메인 규칙인가?

분류 전에 먼저 물어야 할 질문이 있다.

> **"이 규칙은 비즈니스가 이해하고 바꾸자고 말할 수 있는 것인가?"**

| 구분 | 예시 | 처리 |
|------|------|------|
| **도메인 규칙** | 가입 시 상태는 ACTIVE, 중복 가입 불가 | → 아래 4분류로 진행 |
| **기술/보안 규칙** | 비밀번호 해시 저장, JWT 토큰 만료 | → 도메인 규칙표에서 제외. 애플리케이션/인프라 계층에서 처리 |
| **표현/저장 규칙** | JSON 직렬화 형식, DB 컬럼 타입 | → 도메인 규칙표에서 제외. 어댑터 계층에서 처리 |

도메인 규칙이 아닌 것을 규칙표에 넣으면 분류가 흐려진다.

#### 네 가지 분류

0번 질문을 통과한 도메인 규칙은 아래 질문 순서대로 분류한다.

**1단계: 이 규칙은 무엇에 대한 것인가?**

| 분류 | 정의 | 판별 질문 |
|------|------|-----------|
| **값 검증** | 값 자체가 유효한 형태인지 보장 | "단일 값만 보고 판단 가능한가?" |
| **불변식** | 위반하면 애그리거트 자체가 모순이 되는 제약 | 아래 반례 판별법 참고 |
| **정책** | 허용 여부, 대상 판정, 계산 방식을 결정하는 규칙 | "허용/대상/계산 방식을 정하는 건가?" |
| **후속 반응** | 상태 변경 후 이어서 해야 할 일 | "상태가 바뀐 뒤 반드시 이어서 해야 할 일인가?" |

**불변식 판별법: 반례를 만들어본다**

"깨진 상태"를 미리 정의하는 게 아니라, **규칙을 부정해서 반례를 만들고** 그 반례를 허용해도 괜찮은지 묻는다.

```
1. 규칙을 구체 문장으로 쓴다
2. 그 규칙이 거짓인 상태 또는 전이를 만든다 (반례)
3. 그 반례를 허용해도 애그리거트가 정상인지 본다
```

반례에는 두 종류가 있다:
- **반례 상태**: 규칙을 위반한 상태로 저장된 애그리거트 → "이 상태로 저장된 애그리거트를 정상이라고 볼 수 있는가?"
- **반례 전이**: 규칙을 위반한 상태 전이 → "이 전이를 허용해도 도메인 흐름이 일관적인가?"

판단 결과:
- 정상 아님 → **불변식**
- 정상인데 이번 액션만 거절 → **정책**
- 정상인데 뒤에 해야 할 일이 빠짐 → **후속 반응**

#### 같은 최종 상태라도 별도 행위인가?

같은 최종 상태로 가더라도 아래 중 하나라도 다르면 별도 행위 후보로 남겨둔다.

- Actor가 다르다
- 출발 상태가 다르다
- 비즈니스 용어가 다르다
- 감사/사유/후속 반응 의미가 다르다

예: "본인 탈퇴"(ACTIVE→DEACTIVATED)와 "관리자 탈퇴 처리"(SUSPENDED→DEACTIVATED)는
최종 상태만 같을 뿐 Actor, 출발 상태, 용어가 모두 다르므로 별도 행위다.

**2단계: 구현 배치 결정**

의미 분류가 끝나면, 구현을 어디에 둘지는 별도로 판단한다.

| 의미 분류 | 구현 위치 후보 |
|-----------|---------------|
| 값 검증 | Value Object 생성자 |
| 불변식 (단일 애그리거트 내) | 엔티티 내부 |
| 불변식 (다른 애그리거트/저장소 필요) | 도메인 서비스 또는 애플리케이션 서비스 |
| 정책 | aggregate 상태+입력으로 닫히면 엔티티 내부 / 외부 데이터 필요 시 도메인 서비스 / 재사용 계산 규칙이면 Policy 객체 |
| 후속 반응 | 같은 일관성 경계 내 즉시 처리면 직접 호출 또는 애플리케이션 서비스 / 경계를 넘거나 비동기 필요 시 도메인 이벤트 + 핸들러 |

> **주의**: 정책으로 분류되었다는 이유만으로 aggregate 메서드를 하나로 합치지 않는다.
> 규칙의 의미 분류, 실제 검사 위치, aggregate가 외부에 드러내는 행위 API는 별도 결정이다.

> **주의**: 의미상 불변식이지만 검사에 외부 조회가 필요한 경우가 있다.
> 예: "소셜 계정 중복 가입 불가"는 의미상 불변식이지만, 판단에 저장소 조회가 필요하다.
> 이때 "엔티티 안에서 안 닫히니까 불변식이 아니다"가 아니라, **의미는 불변식이고 구현은 서비스에 배치**하는 것이다.

<details>
<summary>하나의 필드에 여러 분류가 공존하는 경우</summary>

닉네임을 예로 들면, 하나의 필드에 여러 분류가 붙을 수 있다.

```
닉네임 변경 요청
│
├─ (1) 값 검증 (Value Object 생성자)
│    "2~20자, 특수문자 불가"
│    → Nickname 생성 시 즉시 검증
│
├─ (2) 불변식 (도메인 서비스)
│    "닉네임 중복 불가"
│    → 다른 Member 조회가 필요하므로 단일 객체 안에서 못함
│
├─ (3) 정책 (Policy 객체)
│    "비속어 포함 금지"
│    → 금지 목록이 바뀔 수 있으므로 교체 가능한 구조
│
└─ (4) 후속 반응 (도메인 이벤트)
     "닉네임 변경 시 활동 로그 기록"
     → 닉네임 변경 이벤트 발행 후 핸들러에서 처리
```

**검증 순서 원칙: 비용이 싼 것부터**

| 순서 | 분류 | 이유 |
|------|------|------|
| 1 | 값 검증 | 가장 싸다 — 객체 하나로 판단, DB 안 침 |
| 2 | 정책 | 메모리 내 판단 가능, DB보다 쌈 |
| 3 | 불변식(중복 등) | 가장 비싸다 — DB 조회 필요 |
| 4 | 후속 반응 | 이벤트 발행 — 위 검증이 다 통과한 후 |

20자 넘는 닉네임 때문에 DB를 찌를 필요가 없다.

</details>

<details>
<summary>심화: 기본 4분류로 애매할 때</summary>

기본 4분류로 분류가 잘 안 될 때 아래 질문을 추가로 던진다.
단순한 도메인에서는 이 섹션을 참고할 필요 없다.

**정책 세분화**

| 유형 | 판별 질문 | 예시 |
|------|-----------|------|
| 판정형 (Specification) | "이 대상이 조건을 만족하는가?" | 이 회원이 쿠폰 대상인가? |
| 계산/선택형 (Strategy) | "어떤 방식/공식/전략을 쓰는가?" | 할인액을 어떤 공식으로 계산하는가? |

**보조 질문**

| 질문 | 답변 선택지 |
|------|-------------|
| 이 규칙을 판단하려면 무엇을 봐야 하나? | 값 하나 / 같은 애그리거트 / 다른 애그리거트 / 현재 시간 / 외부 시스템 |
| 언제 평가되는가? | 생성 시 / 상태 전이 시 / 판정 시 / 이벤트 후 / 배치 |
| 실패하면 어떤 의미인가? | 예외로 막아야 함 / 대상 아님 / 다른 경로 대체 / 재시도 필요 |
| 운영자 설정으로 바뀔 수 있는가? | 고정 / 운영자 설정 가능 / 캠페인별 변경 가능 |

이 질문들은 쿠폰 발급 전략처럼 시간/캠페인/회원등급/재고/중복발급 조건이 섞이는 도메인에서 유용하다.

</details>

> **핵심**: 규칙의 분류 기준은 **대상이 아니라 규칙의 성격**이다.
> 같은 필드의 규칙이라도 성격에 따라 분류와 구현 위치가 달라진다.

### 1-4. 이벤트에서 상태 도출

이벤트 흐름을 보면 상태 enum이 자연스럽게 나온다.

```
가입 → ACTIVE
ACTIVE → SUSPENDED (관리자가 정지)
ACTIVE → DEACTIVATED (본인이 탈퇴)
SUSPENDED → DEACTIVATED (관리자가 탈퇴 처리)
```

- DEACTIVATED는 최종 상태. 다른 상태로 전이 불가.
- 같은 최종 상태라도 출발 상태와 Actor가 다르면 별도 전이 경로다.

> 모든 이벤트가 상태 enum과 1:1은 아니다. "회원 정보가 수정됨"은 이벤트지만 별도 상태가 아닐 수 있다.

## 2단계: 애그리거트 내부 설계

### 2-1. 커맨드별 설계 워크시트

각 커맨드를 설계할 때, 먼저 **1단계 산출물에서 맥락을 수집**한 뒤 설계를 도출한다.
커맨드별로 아래 워크시트를 채운다.

#### 맥락 수집 (1단계에서 가져오기)

1단계의 매핑표, 규칙 분류표, 상태 도출 결과에서 해당 커맨드와 관련된 정보를 한 곳에 모은다.

| 항목 | 출처 | 설명 |
|------|------|------|
| **Actor** | 매핑표 | 누가 이 행위를 하는가 |
| **Event** | 매핑표 | 이 행위의 결과로 무슨 일이 일어나는가 |
| **상태 전이** | 상태 도출 결과 | 상태가 변하는가, 어떻게 변하는가 |
| **관련 값 검증** | 규칙 분류표 | 이 행위의 입력에 적용되는 값 검증 |
| **관련 불변식** | 규칙 분류표 | 이 행위와 관련된 불변식 |
| **관련 정책** | 규칙 분류표 | 이 행위와 관련된 정책 |
| **후속 반응** | 규칙 분류표 | 이 행위 후 이어지는 작업 |

#### 설계 도출 (2단계에서 결정하기)

맥락이 모인 상태에서 아래를 도출한다.

| 도출 항목 | 설명 |
|-----------|------|
| **대상 식별자** | 어떤 애그리거트를 조회/선택하는가 (신규 생성 시 없음) |
| **도메인 입력** | 행위 판단에 실제로 필요한 값 (메서드 파라미터 후보) |
| **상태 후보** | 도메인 입력 중 애그리거트가 보존할 값 |
| **호출 컨텍스트** | 요청자, 권한 등 (인증 계층에서 처리되는 외부 정보) |
| **행위** | 이 커맨드가 애그리거트의 어떤 행위가 되는가 |

> **대상 식별자와 메서드 파라미터의 구분**: 대상 식별자는 애플리케이션 서비스에서 애그리거트를 조회할 때 소비되며, 애그리거트 메서드 파라미터에는 보통 들어가지 않는다. 예를 들어 "회원 정지"에서 memberId는 서비스가 Member를 찾는 데 쓰이고, `member.suspend()`에는 전달되지 않는다.
>
> **호출 컨텍스트 승격 판단 원칙**: 애그리거트가 자기 규칙을 스스로 지킬 수 있어야 한다.
> 어떤 규칙이 애그리거트의 도메인 규칙이라면, 그 규칙을 검증하는 데 필요한 값은 애그리거트가 받아야 한다.
> 서비스가 대신 검증해주는 구조는 애그리거트를 상태만 바꾸는 껍데기로 만든다.
> 단, 애그리거트가 알 수 없는 것(인증/인가 체계 등)은 서비스가 보장한 뒤 호출하는 구조로 둔다.
>
> **승격 기준**: 인증/권한 시스템 자체(SecurityContext, JWT, 세션)는 애그리거트 책임이 아니다.
> 하지만 아래에 해당하면 도메인 입력으로 승격될 수 있다.
> - 요청자 식별자 비교가 전이 허용 조건인 경우 (예: requesterId == memberId)
> - 행위 주체 유형이 전이 경로를 갈라놓는 경우 (예: 본인 탈퇴 vs 관리자 탈퇴 처리)
> - 사유 코드 등이 감사/추적 의미를 바꾸는 경우
>
> 예: "본인만 탈퇴 가능"이 Member의 규칙이면, Member가 `requesterId`를 받아서 직접 검증해야 한다.
> 반면 "관리자만 탈퇴 처리 가능"에서 관리자 여부는 인증 체계의 지식이므로 서비스가 보장한다.
> Member는 자기가 아는 것("SUSPENDED에서만 이 경로 가능")만 직접 지킨다.
>
> 1단계 산출물의 규칙/정책이 행위의 전제 조건과 검증 로직으로 연결된다.

#### 워크시트 통합

모든 커맨드의 워크시트를 채운 뒤:

1. **식별성 결정** — 이 애그리거트를 무엇으로 구분하는가? 이 결정이 있어야 대상 식별자와 메서드 파라미터의 경계가 명확해진다.
2. **통합** — 여러 워크시트에서 반복되는 상태 후보, 행위, 규칙을 합쳐 하나의 애그리거트 초안으로 통합한다.

이 통합 결과가 2-2의 입력이 되고, 최종적으로 2-3의 도메인 모델 요약으로 정리된다.

### 2-2. 내부 구성요소 분류

aggregate 안에서 무엇이 root entity / child entity / value object인지 판단한다.

| 질문 | 예 → | 아니오 → |
|------|------|----------|
| 시간이 지나도 같은 것으로 추적해야 하는가? | entity 후보 | value object 후보 |
| 부모 없이 독립 존재 가능한가? | 별도 aggregate 검토 | child entity |
| 외부에서 직접 접근 필요한가? | 별도 aggregate 검토 | child entity |
| 같은 값이면 교환 가능한가? | value object | entity |
| 같은 일관성 경계 안에서 함께 변경되어야 하는가? | 같은 aggregate 내부 | 별도 aggregate 검토 |

> 마지막 질문은 단순 도메인에서는 자연스럽게 답이 나오지만,
> 복잡한 도메인에서는 애그리거트 경계를 결정하는 핵심 질문이 된다.

#### Value Object로 감쌀지 판단하기

위 표로 "이건 value object다"라고 판단했더라도, **원시 타입(String 등)으로 둘지 별도 VO 클래스로 감쌀지**는 별개의 판단이다.
아래는 규칙이 아니라 **우선순위 힌트**다.

| 기준 | String으로 충분 | Value Object 검토 |
|------|-----------------|----------------------|
| 검증/정규화 로직 | 없거나 null 체크 정도 | 포맷, 길이, 패턴 등 자체 로직 존재 |
| 동일성 비교 | 단순 equals | 대소문자 무시 등 비교 로직 존재 |
| 도메인 행위 | 없음 | 마스킹, 포맷 변환 등 행위 존재 |
| 타입 혼동 | 낮음 | 같은 타입 파라미터가 나란히 있어 순서 실수 가능 |
| 도메인 언어 | 별도 이름 없음 | 독립된 도메인 용어가 있음 (예: Email, Profile) |

> 위 기준에 하나라도 해당하면 VO를 검토한다.
> 해당 사항이 없으면 원시 타입으로 두고, 나중에 필요할 때 감싸도 늦지 않다.

### 2-3. 도메인 모델 요약 작성

코드 작성 전에 아래 항목을 간단히 정리한다.

```
[애그리거트 이름] — 엔티티 (애그리거트 루트)
  식별성:
    - 무엇으로 구분하는가

  핵심 상태:
    - 애그리거트가 보존하는 상태

  핵심 행위:
    - 외부에 제공하는 행위

  규칙:
    - 불변식 (상태/전이)
    - 정책

  외부 협력 지점:
    - 외부 서비스/저장소가 필요한 규칙

  후속 반응:
    - 상태 변경 후 발행하는 도메인 이벤트
```

<details>
<summary>예시: letterPick의 Member</summary>

```
[Member] — 엔티티 (애그리거트 루트)
  식별성:
    - memberId (PK, 인공키)

  핵심 상태:
    - email (고유 제약), nickname, status

  핵심 행위:
    - register() — 생성
    - changeNickname() — 닉네임 변경
    - suspend() — 정지
    - withdraw() — 본인 탈퇴
    - withdrawByAdmin() — 관리자 탈퇴 처리

  규칙:
    - 가입 시 상태는 ACTIVE (생성 불변식)
    - DEACTIVATED에서 다른 상태로 전이 불가 (전이 불변식)
    - ACTIVE 회원은 본인만 탈퇴 가능 (정책 — 전이 경로 분리)
    - SUSPENDED 회원은 관리자만 탈퇴 처리 가능 (정책 — 전이 경로 분리)
    - ACTIVE에서만 정보 수정 가능 (정책)

  외부 협력 지점:
    - 이메일 중복 검증 → 저장소 조회 필요
    - 닉네임 중복 검증 → 저장소 조회 필요

  후속 반응:
    - (없음)
```

</details>

## 3단계: TDD로 코드 구현

### 3-1. 테스트 백로그 만들기

2-3 도메인 모델 요약의 **핵심 행위, 규칙, 외부 협력 지점**을 입력으로 테스트 백로그를 만든다.
각 항목을 "정상 케이스"와 "위반 케이스"로 쪼개면 테스트 백로그가 된다.
백로그는 시작점이지 고정본이 아니다. 구현 중 발견한 설계 모순과 추가 규칙으로 계속 갱신한다.

- 값 검증 규칙 → 해당 값이 VO라면 VO 테스트로, 원시 타입으로 남긴 값이면 생성/행위 테스트 안에서 검증
- 생성/전이 불변식, 정책 → 애그리거트 테스트
- 외부 협력 지점의 규칙(저장소 조회 필요 등) → 도메인 테스트 범위 밖. 서비스 테스트로 미룬다

<details>
<summary>예시: letterPick Member의 테스트 백로그</summary>

**값 검증 — VO 테스트:**
- Email: 유효한 이메일 생성 / 잘못된 형식 실패
- Nickname: 유효한 닉네임 생성 / 길이·특수문자 위반 실패

**생성 불변식:**
- register() 후 상태는 ACTIVE

**전이 불변식:**
- DEACTIVATED에서 상태 전이 시도 → 실패

**정책 — 행위별 정상 + 실패:**
- withdraw(): ACTIVE 본인 → 성공 / 본인 아님 → 실패 / SUSPENDED에서 시도 → 실패
- withdrawByAdmin(): SUSPENDED → 성공 / ACTIVE에서 시도 → 실패
- changeNickname(): ACTIVE → 성공 / SUSPENDED에서 시도 → 실패
- suspend(): ACTIVE → 성공 / DEACTIVATED에서 시도 → 실패

**도메인 테스트 범위 밖:**
- 이메일 중복 검증 → 서비스 테스트
- 닉네임 중복 검증 → 서비스 테스트

</details>

> **핵심**: 이 백로그에 enum, class 같은 구조물은 없다.
> 도메인 규칙만 있고, 구조물은 테스트를 통과시키기 위해 필요할 때 만든다.

### 3-2. 첫 테스트 고르기

첫 테스트는 enum, VO, entity 같은 구조물 기준으로 고르지 않는다.
아래 세 기준을 가장 잘 만족하는 규칙에서 시작한다.

| 기준 | 설명 | 예 |
|------|------|-----|
| 작다 | 통과시키는 데 만들 것이 적다 | VO 값 검증, 생성 불변식 |
| 의미 있다 | 도메인 규칙 하나를 증명한다 | "가입 시 상태는 ACTIVE" |
| 닫혀 있다 | 외부 의존 없이 완결된다 | VO 생성자 테스트, 생성 규칙 테스트 |

고정 순서는 없다. VO가 없으면 생성 규칙부터, 규칙이 복잡하면 가장 단순한 행위부터 시작해도 된다.

### 3-3. 한 사이클: Red → Green → Refactor

```
(1) 테스트 작성 — 컴파일되지 않아도 좋다 (Red)
(2) 컴파일을 통과시킬 최소 스켈레톤 만들기
    — enum, VO, 엔티티 클래스는 여기서 필요한 만큼만 생긴다
(3) 테스트 통과시키기 (Green)
(4) 테스트 코드와 구현 코드를 리뷰한다 — 불필요한 import, 네이밍, 의도가 맞는지 확인
(5) 전체 테스트 돌리기
(6) 리팩토링 — 이름 정리, 중복 제거, 도메인 예외/guard clause 정리
(6) 백로그에서 다음 테스트 고르기 → (1)로 돌아가기
```

(2)에서 "enum부터", "클래스부터"가 아니라 **"지금 테스트가 요구하는 것부터"** 만든다.

<details>
<summary>예시: letterPick Member 첫 세 사이클</summary>

**사이클 1**: "Email VO — 유효한 이메일 생성"
- Email 클래스가 없다 → 최소 스켈레톤으로 만든다
- 테스트 통과 → 실패 케이스 테스트 추가 → 검증 로직 구현

**사이클 2**: "register() 후 상태는 ACTIVE"
- Member 클래스, MemberStatus enum이 없다 → 만든다
- 테스트 통과

**사이클 3**: "withdraw() — ACTIVE 본인 탈퇴 성공"
- withdraw() 메서드 추가, requesterId 파라미터
- 테스트 통과 → 실패 케이스 추가 (본인 아님, SUSPENDED에서 시도)

</details>

### 3-4. 테스트 습관

- 테스트 하나 성공 → **전체 테스트 돌리기** → 다음 테스트
- 테스트 코드는 **준비 / 실행 / 검증** 사이에 빈 줄로 구분
- 도메인 테스트는 기술 의존 없이 **수십 ms 수준**으로 빠르다
- 반복되어 의도가 흐려지는 생성 로직은 **Fixture 클래스**나 헬퍼 메서드로 분리한다
- 테스트 네이밍: 행위 이름 + 실패 케이스는 **조건과 결과**를 드러낸다 (예: `withdraw()`, `withdrawFailsWhenNotOwner()`)
- `@BeforeEach`에서 기본 상태(정상 생성된 애그리거트)를 준비하면 각 테스트가 간결해진다

### 3-5. 진행 중 판단 기준

| 상황 | 판단 |
|------|------|
| 테스트를 쓰려는데 VO가 없다 | VO가 현재 규칙의 핵심이면 VO 테스트로 내려간다. 아니면 최소 스켈레톤만 추가한다 |
| 행위 테스트 중 상태 전이가 필요하다 | 상태 enum을 이때 만들거나 확장한다 |
| 외부 협력이 필요한 규칙이다 | 도메인 테스트 범위 밖. 서비스 테스트로 미룬다 |
| Fixture가 없어서 준비 코드가 길다 | Fixture 클래스를 만들고 돌아온다 |
| 테스트가 설계 모순을 드러낸다 | 2단계 문서를 되돌아가 수정한다 |

## 실전 체크리스트

### 이벤트스토밍 후

- [ ] 이벤트 → 커맨드 → 애그리거트 매핑표 작성했는가?
- [ ] 정책/규칙 목록을 정리했는가?
- [ ] 0번 질문으로 도메인 규칙과 기술/보안 규칙을 걸러냈는가?
- [ ] 규칙을 값 검증 / 불변식 / 정책 / 후속 반응으로 분류했는가?
- [ ] 의미 분류와 구현 배치를 분리해서 판단했는가?
- [ ] 같은 최종 상태로 가는 커맨드가 여럿이면 별도 행위로 분리했는가?
- [ ] 규칙 목록에 메서드명을 먼저 적지 않았는가? (도메인 문장으로 적었는가?)
- [ ] 이벤트에서 상태와 상태 전이를 도출했는가?

### 애그리거트 설계

- [ ] 커맨드별 워크시트를 채웠는가? (맥락 수집 → 설계 도출)
- [ ] 워크시트에 1단계의 값 검증/불변식/정책/후속 반응이 빠짐없이 연결되었는가?
- [ ] 여러 워크시트의 결과를 하나의 애그리거트 초안으로 통합했는가?
- [ ] 각 구성요소가 root entity / child entity / value object 중 뭔지 판단했는가?
- [ ] VO 판단 힌트에 해당하는 값은 Value Object를 검토했는가?
- [ ] 정책 분류가 메서드 분리 여부를 자동으로 결정하지 않았는가?
- [ ] 호출 컨텍스트에 있던 값 중 승격이 필요한 것을 검토했는가?
- [ ] 도메인 모델 요약을 작성했는가? (식별성, 핵심 상태, 핵심 행위, 규칙, 외부 협력 지점, 후속 반응)

### TDD 구현

- [ ] 2-3 도메인 모델 요약의 규칙에서 테스트 백로그를 뽑았는가?

- [ ] 각 규칙마다 정상 + 실패 테스트를 작성했는가?
- [ ] 반복되는 생성 로직을 Fixture로 분리했는가?
- [ ] 외부 협력이 필요한 규칙을 도메인 테스트에서 제외했는가?
- [ ] 전체 테스트가 통과하는가?
