# 입력 모델 분리 가이드

> 컨트롤러 DTO, 포트 계약 객체, 엔티티 입력을 어디까지 분리할지에 대한 실전 기준.
> 특정 회원 도메인 전용 문서가 아니라, 프로젝트 전반에 적용하는 범용 가이드다.
> 예시는 주로 `member`를 사용하지만, 규칙 자체는 다른 도메인에도 동일하게 적용한다.

---

## 핵심 원칙

입력 모델은 아래 3단계로 생각한다.

1. **컨트롤러 DTO**
   - HTTP 요청 바인딩용 모델
   - JSON 구조, 필드명, 요청 포맷 변화에 대응

2. **포트 계약 객체**
   - 애플리케이션 유스케이스 입력 모델
   - provided 포트가 받는 request/command 객체

3. **엔티티 입력**
   - 엔티티가 알아야 하는 도메인 값
   - 인자가 적으면 직접 받고, 많아지면 도메인 의미가 있는 묶음 객체로 전달

즉 흐름은 다음과 같다.

```text
Controller DTO
 -> Port Request
 -> Entity(도메인 값 직접 or 도메인 묶음 객체)
```

---

## 왜 나누는가

### 컨트롤러 DTO를 따로 두는 이유

- 웹/API 입력 포맷은 유스케이스 계약과 다를 수 있다
- 외부 API 필드명, JSON 구조, 헤더/쿼리/바디 조합은 웹 계층 관심사다
- 웹 요구사항 변화가 포트 계약까지 전파되지 않게 막는다

### 포트 계약 객체를 따로 두는 이유

- provided 포트는 애플리케이션이 외부에 제공하는 유스케이스 계약이다
- 포트 시그니처를 일관되게 유지하기 쉽다
- `@Valid` 같은 입력 계약 검증을 표현하기 좋다

### 엔티티가 request를 직접 알지 않게 하는 이유

- 엔티티는 유스케이스 입력 전체가 아니라 자기 생성/행위에 필요한 도메인 값만 알아야 한다
- request 구조 변화가 엔티티까지 직접 전파되는 것을 줄인다
- 생성 책임을 엔티티 쪽에 남기기 쉽다

---

## 기본 권장안

이 프로젝트에서는 아래 기준을 기본값으로 삼는다.

### 1. 컨트롤러는 웹 전용 DTO를 사용한다

예:

```java
public record RegisterHttpRequest(
        String field1,
        String field2
) {
}
```

### 2. provided 포트는 유스케이스별 request 객체 하나를 받는다

예:

```java
public interface SomeUseCase {
    Result execute(@Valid SomeRequest request);
}
```

여기서 말하는 “포트 request로 통일”은
메서드 인자를 없앤다는 뜻이 아니라,
`String, Long, boolean...`처럼 흩어진 인자 대신 유스케이스 입력을 request 객체 하나로 받는다는 뜻이다.

### 2-1. 초기 단계에서는 컨트롤러 DTO와 포트 request를 공유할 수 있다

초기 TDD 단계에서 아래 조건이면 하나의 `Request` 객체를 컨트롤러와 포트가 같이 사용해도 된다.

- 웹 입력과 유스케이스 입력이 거의 동일하다
- 빠르게 구현하고 검증하는 것이 중요하다
- 아직 웹 전용 필드나 외부 API 호환성 이슈가 없다

이때의 기준은 다음과 같다.

```text
Controller DTO == Port Request
Entity 입력은 분리
```

즉 허용되는 공유 범위는 컨트롤러와 포트까지이며,
엔티티는 여전히 request 객체를 직접 받지 않는다.

나중에 아래 신호가 보이면 포트 전용 `Command`를 분리한다.

- 컨트롤러 요청 필드가 웹 전용으로 늘어난다
- API 포맷과 유스케이스 입력이 달라진다
- 다른 어댑터가 같은 포트를 호출한다
- 포트 계약을 외부 API와 독립적으로 유지하고 싶어진다

### 3. 엔티티는 request를 직접 받지 않는다

엔티티는 다음 둘 중 하나를 사용한다.

- 인자가 적으면 도메인 값 직접 받기
- 인자가 많으면 도메인 의미가 있는 묶음 객체 받기

예:

```java
Aggregate.create(Value1 value1, Value2 value2)
```

또는

```java
Aggregate.create(AggregateCreation creation)
```

---

## 엔티티 입력 기준

### 인자가 적을 때

도메인 값 객체를 직접 받는다.

```java
Aggregate.create(Value1 value1, Value2 value2, Value3 value3)
```

이 방식은:

- 엔티티가 필요한 값이 명확하고
- request 구조와 엔티티 경계가 섞이지 않으며
- 생성 규칙을 엔티티 메서드 안에 두기 쉽다

### 인자가 많을 때

request DTO를 그대로 넘기지 말고, 도메인 의미가 있는 묶음 객체로 재구성한다.

예:

```java
Aggregate.create(AggregateCreation creation)
```

여기서 `AggregateCreation`은 단순 request DTO가 아니라,
해당 애그리거트 생성에 필요한 도메인 정보 묶음이어야 한다.

## 빌더 사용 기준

### 엔티티 빌더를 직접 여는 경우 주의점

```java
Member.builder()
    .email(...)
    .nickname(...)
    .status(...)
    .build();
```

이 방식은 외부가 엔티티 내부 상태를 직접 조립하게 만들 수 있다.

주의할 점:

- 어떤 필드가 필수인지 흐려질 수 있다
- 값 조합은 가능하지만 도메인적으로 말이 안 되는 상태가 생길 수 있다
- 생성 시나리오의 의미가 `signup()` 같은 행위보다 약해진다

단, private 생성자나 `build()` 내부에서 생성 시점 검증을 강하게 하면 사용할 수는 있다.
문제는 “빌더를 쓴다” 자체보다, 외부에 엔티티 조립 인터페이스를 얼마나 노출하느냐에 있다.

### 더 나은 대안

- 엔티티 생성은 정적 팩토리 메서드 사용
- 입력이 많으면 Spec/Registration 같은 도메인 묶음 객체에 builder 사용

예:

```java
AggregateCreation creation = AggregateCreation.builder()
        .value1(...)
        .value2(...)
        .build();

Aggregate aggregate = Aggregate.create(creation);
```

이 구조에서는:

- 외부는 입력 묶음만 조립하고
- 엔티티는 최종 생성 책임을 유지한다

## 언제 분리를 생략할 수 있는가

### 컨트롤러 DTO와 포트 request 공유

초기 단계에서 아래 조건이면 공유 가능하다.

- API가 단순하다
- 웹 입력 포맷과 유스케이스 입력이 사실상 동일하다
- 빠르게 진행하는 편이 중요하다

다만 이 경우에도 엔티티까지 그 객체를 직접 넘기지는 않는다.

즉 허용 가능한 타협은 다음까지다.

```text
Controller DTO == Port Request
Entity 입력은 별도
```

### 엔티티 입력을 request 객체로 직접 받는 경우

권장하지 않는다.

이 구조는:

- 유스케이스 입력 구조 변화가 엔티티 구현 변화로 바로 이어지기 쉽고
- 엔티티가 유스케이스 request shape를 알게 되며
- 생성 책임이 흐려질 수 있다

---

## 변경 전파 시나리오

### 분리하지 않은 경우

하나의 request 객체를 컨트롤러와 포트가 같이 쓰면:

- 웹 필드명 변경
- 웹 전용 입력 추가
- API 호환성 대응

같은 변화가 포트 계약까지 같이 전파된다.

### 분리한 경우

컨트롤러 DTO와 포트 request를 분리하면:

- 웹 변화는 컨트롤러 DTO에서 흡수
- 유스케이스 변화는 포트 request에서 흡수
- 엔티티는 도메인 변화가 있을 때만 영향

즉 변경 여파가 경계에서 멈춘다.

---

## 프로젝트 적용 요약

- **컨트롤러**: 웹 DTO 사용
- **provided 포트**: 유스케이스 request 객체 하나로 통일
- **초기 단계 허용 타협**: 컨트롤러 DTO와 포트 request는 하나의 `Request` 객체를 공유할 수 있음
- **애플리케이션 서비스**: 포트 request를 도메인 값/도메인 묶음으로 변환
- **엔티티**: request를 직접 받지 않고, 필요한 도메인 값 또는 도메인 묶음 객체를 받음

이 문서의 목적은 “무조건 다 분리”가 아니라,
경계가 들쭉날쭉해지는 것을 막으면서도 실용적인 수준의 일관성을 유지하는 것이다.

## 작명 기준

작명은 구조만큼 중요하다.
같은 역할의 객체에 이름이 들쭉날쭉하면 경계가 더 빨리 무너진다.

### 기본 원칙

- 같은 유스케이스를 `signup/register/create`처럼 섞지 않고 하나로 통일한다
- 어떤 단어를 쓸지는 도메인별 용어사전과 이벤트스토밍 결과를 따른다
- 역할이 다르면 이름도 달라져야 한다

### 권장 작명

#### 1. 컨트롤러 DTO

초기에는 포트 request와 공유할 수 있으므로 다음 이름을 기본값으로 사용한다.

- `{UseCase}Request`

나중에 컨트롤러 DTO와 포트 request를 분리하면,
컨트롤러 DTO는 `Request`를 유지하고 포트 쪽 이름만 바꾼다.

#### 2. 포트 계약 객체

초기에는 컨트롤러 DTO와 공유하므로 다음 이름을 사용한다.

- `{UseCase}Request`

분리 시점이 오면 포트 계약 객체는 `Command`로 전환한다.

- `{UseCase}Command`

즉 현재 기본 전략은 다음과 같다.

```text
초기: Controller DTO == Port Request == {UseCase}Request
분리 후: Controller DTO = {UseCase}Request / Port Contract = {UseCase}Command
```

#### 3. 엔티티 입력 묶음 객체

엔티티 쪽은 `Request`, `Command`, `Dto` 같은 이름을 쓰지 않는다.
도메인 의미가 드러나는 명사형 이름을 사용한다.

예:

- `{Aggregate}Creation`
- `{Aggregate}Details`
- `{Aggregate}Profile`
- `{Concept}`

이렇게 해야 엔티티가 웹/포트 경계 모델을 직접 아는 구조로 흐르지 않는다.

### member 도메인 예시

위 규칙을 member 도메인에 적용하면 다음처럼 된다.

- 컨트롤러/포트 공유 시: `MemberRegisterRequest`
- 포트 분리 시: `MemberRegisterCommand`
- 엔티티 입력 묶음: `MemberRegistration`

이 예시는 규칙 설명용일 뿐, 다른 도메인에도 같은 방식으로 적용한다.
